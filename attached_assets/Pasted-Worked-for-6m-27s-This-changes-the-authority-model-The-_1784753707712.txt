This changes the authority model

The problem is not simply that a few QuickBooks names are wrong. It is that the system has been treating a QuickBooks incoming-money row as though all of its attributes describe the same real-world thing:

amount + date + bank arrival + donor + gift + coding

They do not.

This also reveals a second issue: the individual QBO entries often use the net Stripe amount, while the donor’s gift is the gross charge amount. So QBO may be wrong both about the donor and about the appropriate gift grain.

Authority should be defined by attribute

No single system is authoritative for the entire transaction.

Fact	Authoritative source
Money reached the bank	Bank-side/QBO deposit amount, date, and account
A particular Stripe payout produced that deposit	Stripe payout plus confirmed settlement link
Which charges were in the payout	Stripe payout ID and Stripe balance transactions
Gross gift amount	Stripe charge
Stripe fee	Stripe charge/balance transaction
Refund or dispute	Stripe
Processor-supplied payer information	Stripe/Donorbox metadata
Canonical donor identity	CRM resolution, supported by processor information
Gift restrictions and allocations	CRM gift/pledge records and supporting documentation
What QuickBooks currently says	Raw QBO transaction fields
What QuickBooks should have said	CRM-derived coding plus finance review

The key distinction is:

QuickBooks is authoritative evidence of the accounting record and bank settlement. It is not necessarily authoritative evidence of donor identity or processor-batch composition.

The three relationship planes

Stripe payout
    ↓ settlement link
QBO/bank deposit

This relationship says:

These are the same net dollars at the processor and bank grains.

Historical repair process

For all historical Stripe transactions, I would rebuild truth from the processor outward.

For each Stripe payout
Read the exact charge membership from stripe_payout_id.
Verify that charge net totals equal the payout.
Match the payout to the bank/QBO deposit using net amount, date, account, and trace information.
Confirm the settlement link regardless of QBO payer labels.
Link or mint one CRM gift per real Stripe charge using the gross charge amount.
Resolve donor identity from Stripe/Donorbox and CRM—not from QBO.
Remove or demote any counted QBO-to-gift applications representing the same processor money.
Record QBO donor/coding differences as historical discrepancies.
Leave QuickBooks unchanged.

The settlement model already recognizes that a confirmed payout/deposit link and its constituent charges represent the same dollars at different grains, and that coarse QBO applthe charge-level applications. fileciteturn322file0L36-L40

Given the apparent shifted-name pattern, I would run this across all historical Stripe payouts, not merely repair the remaining unmatched cases individually.
