Assessment

The central problem is not insufficient documentation. It is that the documentation no longer has a reliable hierarchy.

Agents currently encounter:

A 336-line replit.md combining permanent principles, volatile implementation details, operational troubleshooting, feature descriptions, and test benchmarks.
208 memory files totaling roughly 80,000 words.
A flat MEMORY.md with 84 entries, where canonical architecture and one-off incidents receive equal prominence.
36 memory files with no incoming link from the memory index or another memory file.
Several incompatible descriptions of the same system state.

The uploaded replit.md, for example, still says that gift type derives loan_or_grant and that quickbooks_tie_status is persisted and maintained by applyGiftQbTieMany, although both were subsequently retired. It also includes exact test counts and timing measurements that will inevitably become stale.

Most important findings
1. replit.md should be an operating contract, not a system encyclopedia

Its strongest content is:

Contract-first development
Header-plus-allocation modeling
Derived-status discipline
Donor XOR
Production safeguards
Architecture-before-patching
Reduction as the success criterion

Those belong in replit.md.

These do not:

Exact passing-test counts
Historical performance measurements
Detailed feature descriptions
Migration-phase status
Names of transitional fields and functions
Long dedicated-test-database implementation details

Those should be linked documents.

2. MEMORY.md should route, not catalog

The current file encourages an agent to scan dozens of highly specific notes before understanding which ones remain authoritative.

It also contains summaries that disagree with their linked files. One prominent example says gift type is still live, while the linked memory file says the column was dropped.

The replacement index groups memory into:

Canonical orientation
Domain indexes
Core data-model invariants
Delivery and testing
Frontend conventions
High-risk operational notes
Historical context
3. Reconciliation documentation needs three distinct layers

At present, reconciliation-design.md mixes:

Normative architecture
Obsolete “current state”
Migration history
Abandoned UI designs
Current cluster-workbench decisions

It contains statements that the six-queue workbench is the accepted final design and other statements that the cluster workbench supersedes it.

The clean division should be:

Business rules: what the system must mean.
Target architecture: how those rules should be represented.
Current implementation status: what is actually implemented and where it drifts.

workbench-business-rules.md is the strongest normative document and should remain so. The new proposed reconciliation-status.md prevents agents from treating current bugs as intended behavior.

4. Memory lacks a retirement policy

Many notes describe fields, routes, and workbench surfaces that no longer exist. All 207 topic files have frontmatter, but almost none have an explicit machine-readable status such as:

status: current | transitional | historical
last_verified: 2026-07-21

The proposed policy requires:

Current code and test anchors
A retirement condition for transitional notes
Historical files under legacy/ or archive/
Reachability from a domain index
Same-change cleanup when a field or architecture is retired
